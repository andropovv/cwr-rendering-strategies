# Исследование адаптивных стратегий рендеринга и выборочной гидратации

Четыре конфигурации Next.js с общими компонентами и данными для сравнения влияния на Core Web Vitals (LCP, INP, CLS).

## Структура

- **packages/shared** — общие компоненты (Layout, BlogCard, ProductPageContent, StatsCard, ChartBar, ActivityList), mock-данные (блог, товары, дашборд), стили, интерактивные клиентские компоненты (`DashboardInteractive`, `ProductControls`).

## Конфигурации и их различия

### 1. **apps/ssr-full** — SSR + полная гидратация

**Стратегия рендеринга:**
- Данные запрашиваются на сервере (async Server Components).
- Вся страница передаётся в **единый клиентский компонент** (`DashboardClient`, `ProductPageClient`), который гидратируется целиком.
- Интерактивность: состояние хранится в памяти клиента (`useState`), обновления происходят без запросов к серверу.

**Особенности:**
- Максимальный объём клиентского JavaScript (вся интерактивная логика гидратируется).
- Быстрая интерактивность после гидратации (нет задержек на серверные запросы).
- Подходит для сравнения как «базовая линия» с полной гидратацией.

**Пример:** `/dashboard` — весь дашборд (фильтры, график, список) в одном клиентском компоненте `DashboardInteractive`.

---

### 2. **apps/streaming** — Streaming SSR + прогрессивная гидратация

**Стратегия рендеринга:**
- HTML отправляется **потоком** (chunks) через `Suspense`.
- Данные загружаются асинхронно на сервере, части страницы рендерятся по мере готовности.
- Клиентский компонент (`DashboardInteractive`) оборачивается в `Suspense` с fallback, гидратация начинается после получения данных.

**Особенности:**
- **LCP улучшается** за счёт раннего отображения частей страницы (не нужно ждать все данные).
- Гидратация начинается позже, но пользователь видит контент раньше.
- Интерактивность та же, что в ssr-full (клиентское состояние), но гидратация может быть отложена.

**Пример:** `/dashboard` — `Suspense` вокруг `DashboardBlock`, который стримит `DashboardInteractive` после загрузки данных.

---

### 3. **apps/rsc-selective** — RSC + выборочная гидратация

**Стратегия рендеринга:**
- **Большая часть страницы — Server Components** (StatsCard, ChartBar, ActivityList рендерятся на сервере, не гидратируются).
- Только **один маленький клиентский островок** (`ControlsClient`) гидратируется для интерактивности.
- Интерактивность: клиентские контролы обновляют URL (`searchParams`), сервер перерисовывает страницу с новыми данными.

**Особенности:**
- **Минимальный клиентский JS** — гидратируется только блок контролов (~2.5 kB для `/dashboard`).
- При клике на фильтр происходит навигация (обновление URL), сервер пересчитывает данные и отправляет новый HTML.
- **INP может быть выше** из-за навигации, но **размер JS значительно меньше** (выборочная гидратация работает).

**Пример:** `/dashboard` — серверные `StatsCard`, `ChartBar`, `ActivityList` + маленький `ControlsClient`, который обновляет URL.

---

### 4. **apps/islands** — Islands (гибрид)

**Стратегия рендеринга:**
- Почти всё — **Server Components** (статический HTML).
- Интерактивность разбита на **несколько маленьких изолированных островков** (`RangeIsland`, `MetricIsland`, `SortIsland`), каждый гидратируется независимо.
- Каждый островок обновляет URL, сервер перерисовывает страницу.

**Особенности:**
- **Ещё меньше клиентского JS** — три маленьких островка вместо одного блока контролов.
- Каждый островок гидратируется независимо (можно отложить гидратацию менее критичных).
- Архитектура ближе к Astro/Qwik (минимальная гидратация, максимум серверного рендеринга).

**Пример:** `/dashboard` — три отдельных клиентских компонента (`RangeIsland`, `MetricIsland`, `SortIsland`) + серверные карточки/график/список.

---

## Ключевые различия для измерения INP

| Конфигурация | Клиентский JS (dashboard) | Гидратация | Интерактивность |
|--------------|---------------------------|------------|-----------------|
| **ssr-full** | ~90 kB (весь дашборд) | Полная | Клиентское состояние (мгновенно) |
| **streaming** | ~89 kB (весь дашборд) | Прогрессивная (после стрима) | Клиентское состояние (мгновенно) |
| **rsc-selective** | ~89.7 kB (только ControlsClient) | Выборочная (только контролы) | Навигация (обновление URL) |
| **islands** | ~89.9 kB (три островка) | Минимальная (три островка) | Навигация (обновление URL) |

**Важно:** В `rsc-selective` и `islands` интерактивность работает через обновление URL (`router.replace`), что может влиять на INP (навигация vs. мгновенное обновление DOM). Это **намеренное различие** для демонстрации разных стратегий гидратации.

## Запуск

```bash
# Установка зависимостей (из корня монорепо)
npm install

# Сборка общего пакета
npm run build -w @cwr/shared

# Разработка по приложениям (разные порты)
npm run dev:ssr-full      # http://localhost:3001
npm run dev:streaming     # http://localhost:3002
npm run dev:rsc-selective # http://localhost:3003
npm run dev:islands       # http://localhost:3004
```

## Страницы в каждом приложении

- **/** — главная с описанием конфигурации и ссылками.
- **/blog** — список постов блога.
- **/blog/[slug]** — страница поста.
- **/product/[id]** — страница товара.
- **/dashboard** — дашборд (метрики, график, активность).
